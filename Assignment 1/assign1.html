2. Assignment Instructions

As the assignment will be auto-graded, it is very important that your submitted answers conform to the following requirements.

  1. For each question, you are to write a single "create or replace view" SQL statement to answer the question. You MUST use the view schema provided for each question without changing any part of the view schema (i.e., view name, column names, or the order of the columns). For example, the provided view schema for Question 1 is "v1 (sid, cid)", and if your answer to this question is the query "select distinct sid, cid from tutors", then you could write your answer as follows:

    	create or replace view v1 (sid, cid) as

	from tutors;

  2. Each "create or replace view" statement must terminate with a semicolon.

  3. Each answer must be a syntactically valid SQL query without any typographical errors: a SQL answer that is syntactically invalid or contains very minor typographical error will receive 0 marks even if the answer is semantically correct.

  4. For each question, your answer view must not contain any duplicate records.

  5. You must not enter any extraneous text for your answer (e.g., "My answer is: create or replace view v1 (sid, cid) as select distinct sid, cid from tutors;") or enter multiple answers (i.e., submit multiple SQL statements).

  6. Each question must be answered independently of other questions: the answer for a question must not refer to any other view that is created for another question.

  7. You are allowed to create an answer view using a single CTE statement (defining possibly multiple temporary tables). If your answer uses a CTE statement to define temporary table(s), you must not use any of the following table names:

    • Names that are used by the application’s database schema (e.g., Students, Courses)

    • Names that are used by the 10 answer views: v1, v2, …​, v9, v10.

    • Any table name that begins with the prefix zz.

  8. If your answer uses the CTE statement, you must define at most 5 temporary tables for that query.

  9. Your answers must use only the SQL constructs that are covered in the lectures (e.g., window functions are not allowed).

  10. Your answers must be executable on PostgreSQL 14.1 or 14.2.

  11. It is not necessary to write comments in your answer. If you choose to write comments, you should not use SQL comments that begin with two hyphen characters to avoid a potential copy-and-paste problem that results in a query being copied over as a single line with SQL code being commented out (e.g., "create or replace view v1(sid, cid) as select distinct sid, cid — yada yada from tutors;"). Using C-style SQL comments are fine (e.g., "create or replace view v1(sid, cid) as select distinct sid, cid /* yada yada */ from tutors;").

  12. The following are four examples of invalid answers for Question 1:

5. Database Schema

The questions are based on a university database schema that consists of the following eight tables.

  • Departments record information about the departments in the university. Each department has a unique identifier did and belongs to a faculty (specified by faculty).

  • Students record information about students. Each student has a unique identifier sid and belongs to a specific department (specified by did). The attribute year records the year that the student was admitted.

  • Profs record information about professors. Each professor has a unique identifier pid and belongs to a specific department (specified by did).

  • Courses record information about the available courses. Each course has a unique identifier cid, the number of credits, and is offered by a department (specified by did).

  • Offerings record information about the courses (identified by cid) that have been offered. Each course offering is for a session that is defined by the year and semester that it is offered. Thus, a course offering is defined by (cid, year, semester).

  • Teaches record information about the professors (identified pid) who taught in the course offerings (identified by cid, year, semester) and the number of the professor’s teaching hours for that offering (specified by the attribute hour).

  • Tutors record information about the students (identified by sid) who served as teaching assistants in the course offerings (identified by cid, year, semester) and the number of the tutor’s teaching hours for that offering (specified by the attribute hour).

  • Transcripts record the marks (specified by marks) obtained by students (identified by sid) for the course offerings (identified by cid, year, semester) enrolled by the student.

The following are additional constraints on the database:

  • For each course offering in Offerings, there must be at least one professor who taught that course offering, and there must be at least one student who enrolled in that course offering.

  • A student is allowed to enroll in the same course more than once (i.e., in multiple sessions) only if the student has obtained below 50 marks for all the student’s previous enrollments of that course.

  • A student can’t enroll in a course offering in a year before the student’s year of admission.

  • A student can’t tutor a course offering in a year before the student’s year of admission.

  • The domain of credits is the set {2,3,4,5}.

  • The domain of year is the set {1980, 1981, …​}.

  • The domain of semester is the set {1,2}.

  • The domain of marks is the set {0,1, …​,100}.

  • The domain of hours is the set of positive integer values.

The following is the SQL database schema to be used for this assignment.

CREATE TABLE Departments (
	did 		text PRIMARY KEY,
	faculty 	text NOT NULL
);

CREATE TABLE Students (
	sid 		text PRIMARY KEY,
	did			text NOT NULL REFERENCES Departments,
	year		integer NOT NULL CHECK (year >= 1980)
);

CREATE TABLE Profs (
	pid 		text PRIMARY KEY,
	did			text NOT NULL REFERENCES Departments
);

CREATE TABLE Courses (
	cid 		text PRIMARY KEY,
	did			text NOT NULL REFERENCES Departments,
	credits 	integer NOT NULL CHECK (credits in (2,3,4,5))
);

CREATE TABLE Offerings (
	cid 		text REFERENCES Courses,
	year 		integer CHECK (year >= 1980),
	semester 	integer CHECK (semester in (1,2)),
	PRIMARY KEY 	(cid, year, semester)
);

CREATE TABLE Teaches (
	pid 		text REFERENCES Profs,
	cid 		text,
	year 		integer,
	semester 	integer,
	hours		integer NOT NULL CHECK (hours > 0),
	PRIMARY KEY 	(pid, cid, year, semester),
	FOREIGN KEY 	(cid, year, semester) REFERENCES Offerings
);

CREATE TABLE Tutors (
	sid 		text REFERENCES Students,
	cid 		text,
	year 		integer,
	semester 	integer,
	hours		integer NOT NULL CHECK (hours > 0),
	PRIMARY KEY 	(sid, cid, year, semester),
	FOREIGN KEY 	(cid, year, semester) REFERENCES Offerings
);

CREATE TABLE Transcripts (
	sid 		text REFERENCES Students,
	cid 		text,
	year 		integer,
	semester 	integer,
	marks 		integer NOT NULL CHECK (marks >= 0 AND marks <= 100),
	PRIMARY KEY 	(sid, cid, year, semester),
	FOREIGN KEY 	(cid, year, semester) REFERENCES Offerings
);

11. Psql’s quiet mode

As the default mode of psql is quite verbose (it echos each of the executed commands), you can configure psql to be quieter by executing the psql commands in psql-quiet.txt. The execution of psql-quiet.txt needs only be done once for each psql session.

assign1=# \i psql-quiet.txt

On MacOS/Linux, you could setup psql’s configuration file as follows:

$ cp psql-quiet.txt ~/.psqlrc
